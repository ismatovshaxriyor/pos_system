# cloud_server/tenants — guidance for Claude Code

This file loads automatically when working with files under `cloud_server/tenants/`. See the repo-root `CLAUDE.md` for the rest of the system (Bola-side licensing, `cloud_server/sync`, etc.).

### Cloud tenancy (`cloud_server/tenants/models.py`)
`Restaurant`, `License` (one-to-one), `RestaurantStatus` (one-to-one, latest heartbeat metrics — overwritten, not historized), `RemoteCommand` (FK, queue with `pending`/`sent`/`acknowledged`/`completed`/`failed` lifecycle), `ErrorLog` (FK, received via the error-reporting channel described in the repo-root `CLAUDE.md`'s Licensing section), and `SyncedOrder`/`SyncedOrderItem`/`SyncedPayment` (FK, denormalized sales copies received via `POST /api/sync/orders/` — see `local_server/core/CLAUDE.md`'s Sync-ready data model section; **not yet registered in `tenants/admin.py`**, so invisible in the admin until someone adds ModelAdmins). `RestaurantAdmin` shows a green/red Onlayn/Oflayn badge (`is_online`) and has bulk actions to enqueue `block_system`/`unblock_system`/`force_license_renew`/`update_app` commands from the restaurant list. **Fleet update in one action:** `action_release_version` ("⬆️ Tanlangan Bolalarga YANGI VERSIYA yuborish") renders an intermediate page (`templates/admin/release_version.html`) that asks for a version once, then for every *active* selected restaurant sets `desired_version` **and** enqueues an `update_app` command — the one-click counterpart to the older `action_update_app` (which instead reads each restaurant's already-set `desired_version` and blank-skips). See repo-root `CLAUDE.md`'s Update rollout for the full release flow (VERSION file → `release-local-server.yml` builds/pushes the ghcr image → this action → Watchtower).

**Online/offline tracking is deliberately write-cheap** (a fleet of restaurants heartbeating every 60s would otherwise mean constant DB writes): each heartbeat only refreshes a Redis key (`restaurant_metrics_<id>`, TTL 180s); `RestaurantStatus` + `Restaurant.last_seen`/`is_online` are written at most every ~5 minutes (or on an offline→online transition). `tenants.tasks.mark_offline_restaurants` (Celery beat, every 60s) marks a restaurant offline only when **both** signals are gone — no cache key **and** `last_seen` older than 3 minutes — so a Redis restart/flush can't flip the whole fleet to Oflayn (the `last_seen` fallback holds it online until real staleness).

`License` is never created by hand in the admin — `tenants/signals.py::create_default_license` (`post_save` on `Restaurant`, `created=True` only) auto-creates one with a short default `expires_at` (end of the current month + 2 days' grace, `compute_default_license_expiry()`), so onboarding a restaurant means creating the `Restaurant` row and immediately having an activatable key.

**Billing/renewal surfacing:** `RestaurantAdmin.list_display` includes a `license_status_badge` column (red "Muddati o'tgan" if past `expires_at`, orange "N kun qoldi" inside `LICENSE_WARNING_DAYS` = 7, green date otherwise) plus a `LicenseExpiryFilter` (`SimpleListFilter`, buckets: expired/expiring-soon/healthy) in `list_filter`, so restaurants needing payment surface directly in the main list an operator already works from, rather than a separate dashboard page/template. The "To'lov qilindi" bulk action (`RestaurantAdmin.action_mark_paid`, and the equivalent `LicenseAdmin.mark_paid` for operators working from the License list instead — both call the shared `_mark_paid()` helper) resets `expires_at` to a fresh `compute_default_license_expiry()` call; it deliberately only touches `expires_at`, never `is_active` — a restaurant killed via `block_system` for a reason other than non-payment isn't silently unblocked by marking its license paid.
