# local_server/core — guidance for Claude Code

This file loads automatically when working with files under `local_server/core/`. It was split out of the repo-root `CLAUDE.md` (see that file's Architecture section for the parts of the system that live outside this app — licensing/kill-switch, cloud tenancy, etc.).

### Sync-ready data model (`local_server/core/models.py`)
Every local model inherits an abstract `BaseModel`:
```python
class BaseModel(models.Model):
    sync_uuid = models.UUIDField(default=uuid.uuid4, editable=False, db_index=True, unique=True)
    is_synced = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    history = HistoricalRecords(inherit=True)  # django-simple-history - see note below
```
This is how the design in the repo-root `docs/1_ona_bola_arxitekturasi.md` avoids primary-key collisions when local data is merged into the cloud DB: local rows keep normal `BigAutoField` PKs for local use, but carry a globally-unique `sync_uuid` and an `is_synced` flag swept periodically and pushed upstream. **The upstream half of this now exists, for orders only:** `licensing.tasks.sync_completed_orders` (Celery beat, every 60s, `expires=50`) sweeps up to 200 `is_synced=False` orders in a **terminal status** (`completed`/`cancelled` — open orders never leave the Bola) and POSTs them, items + payments included, to Ona's `POST /api/sync/orders/` (`OrderSyncView`, `HeartbeatAuthentication` — so even a blocked restaurant keeps reporting sales). The payload is a denormalized snapshot (`product_name`/`waiter_name` strings, not FKs — cloud copies must survive local menu/staff edits) and **excludes voided `OrderItem`s** (mirroring `Order.total_amount`'s own exclusion — otherwise Ona's summed item total would exceed the order total it's supposed to reconcile against). Ona upserts into `SyncedOrder`/`SyncedOrderItem`/`SyncedPayment` using the Bola-side `sync_uuid` as the *primary key* (`update_or_create` on the order, delete-and-recreate of its child rows) so retried batches are idempotent; Bola flips `is_synced=True` only for the UUIDs Ona echoes back in `synced_uuids`. Downstream sync (Ona → Bola menu push) is still not implemented, and the `force_sync` remote command remains a stub that reports "not implemented yet" back to Ona.

**Order creation is itself idempotent for offline-first clients:** a client may POST `sync_uuid` (a UUID it generated itself) in the body of `POST /api/orders/`; `OrderViewSet.create` looks up an existing `Order` with that `sync_uuid` first and returns it with `200` (not `201`) if found, so a client that retries after a dropped response (spotty restaurant WiFi, not the Bola↔Ona sync above) never opens a duplicate order — a concurrent double-retry race is closed too, by catching the `IntegrityError` from the unique `sync_uuid` constraint and re-fetching. An invalid (non-UUID) `sync_uuid` is rejected with `400` rather than falling through to the DB. Omitting `sync_uuid` entirely still works (falls back to the model default, no idempotency).

**Full field-level history via `django-simple-history`:** `BaseModel.history` puts `HistoricalRecords(inherit=True)` on the abstract base, so every concrete subclass automatically gets its own `Historical<Model>` table (full snapshot per save, not a diff) with zero per-model boilerplate — `inherit=True` is required here or simple_history silently skips subclasses (it warns if you forget it). `request.user` is attached to each historical row via `simple_history.middleware.HistoryRequestMiddleware` in `MIDDLEWARE` (must sit after `AuthenticationMiddleware`). Every registered `ModelAdmin` in `core/admin.py` and `licensing/admin.py` uses `simple_history.admin.SimpleHistoryAdmin` instead of plain `admin.ModelAdmin`, which adds a "History" button/view — don't confuse this with Django's own built-in per-object admin "History" link (visible on `cloud_server` too, since that's core Django, not this package): the built-in one only logs admin-UI edits as a text message, this one captures every DB-level change (API, Celery task, or admin) with the full field state at each version, plus a `diff_against()` helper. Not wired up on `cloud_server` — this is `local_server`-only by design (the "Bola" is where day-to-day operational history matters most).

Local domain models: `User` (custom, `AUTH_USER_MODEL = 'core.User'`, phone number as username validated by a `+998xxxxxxxxx`-style regex, `role` in `manager`/`cashier`/`waiter`; `save()` force-sets `is_staff=True` whenever `role == 'manager'`, so **every** manager account — not just the Ona-provisioned one — logs in with phone+password, no device binding, and is exempt from `generate_registration_code`/PIN login; see Staff PIN + device auth below), `TableZone` (groups tables into areas — Zal/Ko'cha/VIP), `Table` (optional `zone` FK), `Category` (has `image`, optional `printer` FK — see Kitchen printing below), `Product` (`barcode` with `db_index=True`, `image`, plus `cost_price`/`tax_rate`/`is_deleted`), `Order` (`order_type` = dine_in/takeaway/delivery, `tax_amount`, `service_charge`, `note`, `guest_count`), `OrderItem` (kitchen-facing `status` = new/preparing/ready/served, `note`, `modifiers` JSONField, `is_voided`, `is_printed`), `Payment` (`is_voided`, `refunded_of` self-FK, `reference`). **`DELETE /api/products/<id>/` is a soft delete** (`perform_destroy` sets `is_deleted=True` + `is_available=False`): `OrderItem.product` is `on_delete=PROTECT`, so a once-sold product can never be hard-deleted. Soft-deleted products vanish from every API read path (`ProductViewSet.get_queryset`, `BootstrapView`) and can't be added to new orders (`OrderItemSerializer.product_id` queryset excludes them); un-deleting is Django-admin-only. The order lifecycle (`new → in_progress → completed`/`cancelled`) moves **only** through `OrderViewSet` custom actions — `start`, `close`, `cancel`, `add_item`, `add_payment`, `set_discount`: `Order.status` sits in `OrderSerializer.read_only_fields` (as do `tax_amount`/`service_charge` — same client-can't-write-money principle as `discount_amount`), so a generic `PATCH /api/orders/<id>/` can't flip it (that hole previously let anyone mark an unpaid order `completed` — repo-root `future-docs/01_kritik_xatolar.md`), and the standalone `/api/order-items/` ViewSet was removed outright (an `OrderItem` is only ever created via `add_item`). Every mutating action re-checks `Order.status` **inside** `transaction.atomic()` + `select_for_update()` — checking before taking the lock is a TOCTOU race against a concurrent `close`/`cancel`. Don't bypass the actions by writing `OrderItem`/`Payment` rows or `Order.discount_amount` directly from client code:

**Media uploads (`Category.image`/`Product.image`):** `MEDIA_ROOT = BASE_DIR / 'media'`, served at `/media/<path>` via `django.views.static.serve` wired directly into `config/urls.py` — deliberately *not* gated behind `if settings.DEBUG` (unlike Django's usual `static()` helper, which no-ops when `DEBUG=False`), because there's no nginx/whitenoise in front of either compose stack; daphne/runserver serves media itself at any scale this app runs at. `MEDIA_ROOT` matches the `media_data:/app/media` volume already declared in `docker-compose.prod.yml` (no compose change needed to pick this up); dev needs no dedicated volume either since the whole tree is already bind-mounted (`.:/app`) and `media/` is gitignored. **Always pass `context={'request': request}`** to any serializer touching an `ImageField` — without it DRF's `ImageField` renders a relative `/media/...` path instead of an absolute URL, which is why `BootstrapView` (the mobile app's single bulk-fetch endpoint) builds one shared `context` dict up front and passes it to every serializer it calls, not just the ones (`Table`, `Order`) that happened to need it historically.
- `add_item` prices the line server-side from `Product.price` (never trusts a client-supplied price) and persists the client's `note`/`modifiers`; `OrderItem.status` and `is_voided` are serializer-read-only (no kitchen/void API flow exists yet — exposing them writable would let clients pre-void lines). There is no stored total to maintain anymore — `Order.total_amount` is now a computed property (Σ `item.price × quantity` over **non-voided** items), **not a DB column**.
- `add_payment` (any authenticated staff — cashier/waiter/manager) supports **split payments**: an `Order` can have multiple `Payment` rows (`method` = cash/card/other). Rejects amounts that would push total payments above `Order.final_amount`, and rejects any payment on an already `completed`/`cancelled` order. Wrapped in `transaction.atomic()` + `select_for_update()` since multiple cashier terminals could post to the same order concurrently. Persists a client-supplied `reference` (card transaction id); `Payment.is_voided`/`refunded_of` are serializer-read-only until a real void/refund flow exists.
- `close` **requires full payment** (`Order.balance_due == 0`) before it will flip `status` to `completed`, and refuses already-finished orders. `start` (any staff) is the only path `new → in_progress`. `cancel` is manager-gated and refuses `completed` (and already-`cancelled`) orders — money already taken can't be cancelled away. **Gotcha:** `cancel`'s gate lives in `OrderViewSet.get_permissions()` next to `set_discount`'s — that override means `@action(permission_classes=...)` is silently ignored on this ViewSet, so new gated actions must be added to `get_permissions()`, not decorated.
- `set_discount` is gated to managers only (`IsManagerOrAdmin`, via a `get_permissions()` override scoped to that one action), refuses finished orders, and mirrors `ProductViewSet.perform_update`'s existing "notify managers of a financially-sensitive change" pattern (`Notification` + `broadcast_event('discount_applied', ...)`, only when the amount actually changed — fires unconditionally now, there's no exempt "the one real admin" account, see below).
- `Order.total_amount` / `final_amount` / `amount_paid` / `balance_due` are all computed properties (not DB columns), exposed read-only on `OrderSerializer`; `final_amount = max(total_amount - discount_amount + tax_amount + service_charge, 0)`. Both aggregates exclude voided rows — but note the asymmetry in *how*: `amount_paid` filters in SQL (`payments.filter(is_voided=False)`, always a fresh `Sum`, deliberately not cached/prefetched for the concurrent-terminal reason above), while `total_amount` filters in Python (`if not item.is_voided` over `items.all()`) so it rides the ViewSet's `prefetch_related` cache instead of re-querying per order in list views.
- `GET /api/orders/?date=today` (or `?date=YYYY-MM-DD`) lets a waiter/cashier/manager browse order history by day — a day boundary computed against `RESTAURANT_TZ` (`Asia/Tashkent`, hardcoded in `core/views.py`), not Django's global `TIME_ZONE` (UTC): a naive UTC-midnight cutoff would file a ~00:30 local-time order under the wrong day. Combines with (doesn't replace) the existing waiter-scoping filter above, and an unparseable value raises a `400` via DRF's `ValidationError` rather than falling through silently.

### Kitchen printing & routing
`Printer` (name/IP/port — IP/port are unused placeholders, see below) and `PrintJob` (FK to `Printer`+`Order`, `items_snapshot` JSONField, `status` = pending/printed/failed) implement per-category ticket routing: `Category.printer` (optional FK) decides which printer a product's ticket goes to; `services.send_order_to_kitchen(order)` groups an order's not-yet-`is_printed`, non-voided `OrderItem`s by their product's category printer (falling back to any active `Printer`, auto-creating "Asosiy printer (standart)" if none exists at all), creates one `PrintJob` per printer group with a denormalized items snapshot, flips those items' `is_printed=True`, and broadcasts a `new_print_job` WS event. Called from `OrderViewSet.start` (whole order) and from `add_item` when the order is already `in_progress` (so kitchen tickets are incremental — never re-sent for already-printed lines). There is no real thermal-printer/ESC-POS integration yet — despite the `ip_address`/`port` fields on `Printer`, jobs are only ever consumed by whatever's polling `GET /api/print-jobs/` or listening on `ws/events/`, and marked done via `PrintJobViewSet.mark_printed`/`mark_failed` (both just flip `status`, no retry queue). `core/templates/core/kitchen_dashboard.html`, served at `/kitchen/printers/` (station list) and `/kitchen/printers/<id>/` (single-station queue) by `KitchenDashboardView` (plain unauthenticated `TemplateView` — no login screen: the view auto-provisions a permanent `role='cashier'`, `is_staff=False` service account + `StaffDevice` + token and embeds that token/device_id into the rendered page; token exposure via view-source is an acknowledged, accepted risk whose blast radius is deliberately capped at plain-cashier permissions), is a browser-based **virtual printer stand-in** for dev/demo use without real hardware, populated over the same WS channel used everywhere else, with "Chop etildi"/"Xato" buttons calling the mark-printed/mark-failed actions. All waiter-controlled text it renders (item name/note/modifiers, waiter/table names…) goes through its `escapeHtml()` helper before hitting `innerHTML` — keep doing that for any new interpolation, since a `note` containing HTML would otherwise execute on the kitchen screen with the service token's privileges.

### Staff attendance & geofencing
`RestaurantConfig` (singleton, `pk` forced to 1 like `licensing.LicenseState`) holds the restaurant's `latitude`/`longitude` and an `attendance_radius` (meters, default 100) that a manager sets via `RestaurantConfigViewSet` (read open to any authenticated staff, write `IsManagerOrAdmin`-gated). `Attendance` logs one check-in/check-out cycle per row, storing lat/lon at both events for audit. `services.check_in_employee`/`check_out_employee` (`AttendanceViewSet.check_in`/`check_out`, both `POST`, any authenticated staff) reject the request if `RestaurantConfig` isn't set up yet, or if `services.calculate_haversine_distance()` between the submitted coordinates and the restaurant's puts the employee outside `attendance_radius`; check-in additionally refuses a second open (`check_out__isnull=True`) attendance row per user, and check-out refuses when there's no open row. `AttendanceViewSet.get_queryset` scopes to `request.user` unless `role == 'manager'` (managers see everyone's log).

### Staff PIN + device auth and real-time events
There are now **three** ways to obtain a DRF `Token`, a login-method distinction mostly orthogonal to `role`: any `is_staff=True` account logs in with phone + password via the normal `POST /api/auth/login/`, no device binding at all; a `role='waiter'` account can *also* use phone + password, but through a dedicated endpoint that does bind a device (`POST /api/auth/waiter-login/`, TOFU — see below); every other login path is bound to exactly one **device** and unlocks it daily with a 6-digit PIN, never phone+password. This is why `User` carries a separate `pin_hash` field instead of reusing `password` — a leaked PIN would otherwise be replayable through the ordinary login endpoint with no device check at all. **`is_staff` does *not* mean "has more permissions"** — there's no separate "admin" role; every `role='manager'` account (however it logs in) has identical full permissions. `core/permissions.py::IsAdminStaff` keys purely off `role == 'manager'`; `IsManagerOrAdmin`'s write check is still `user.is_staff or user.role == 'manager'` (left as-is by the `IsAdminStaff` role fix — harmless redundancy, but don't copy this OR pattern into new permission classes). **`User.save()` force-sets `is_staff=True` whenever `role == 'manager'`** (not just the single Ona-provisioned account from first activation — every manager created afterward, e.g. by another manager or the `seed_demo_data` command, gets the same treatment automatically), so `is_staff=True` and `role == 'manager'` are effectively the same population; `generate_registration_code` refuses to issue a code for an `is_staff` user (they don't need the PIN/device flow). `is_staff` is otherwise API-invisible by design — `UserSerializer` (`core/serializers.py`) never reads or writes it (also requires `first_name`, unlike the rest of `AbstractUser`'s optional fields) — so a client must use `role` to decide what UI to show, never `is_staff`.

- **Onboarding via admin-issued code:** an admin generates a one-time code (`services.generate_registration_code`, 15 min TTL, `DeviceRegistrationCode`); the staff member redeems it via `POST /api/auth/device/register/` (`DeviceRegisterView`, unauthenticated) with a `device_id` + a PIN they choose (`services.redeem_registration_code`) — this creates the `StaffDevice` row (`is_approved=True` immediately — the admin-issued code *is* the approval), sets `pin_hash`, and returns a token immediately (no separate login step). Daily use is `POST /api/auth/pin-login/` (`PinLoginView`, unauthenticated) with just `device_id` + PIN (`services.verify_pin_login`, which persists `last_login_at` via an explicit `save(update_fields=...)` — it used to only set the attribute in memory and never write it, so the admin panel's last-login column was permanently blank).
- **Onboarding via TOFU + manager approval (waiters only):** `POST /api/auth/waiter-login/` (`WaiterLoginView`, unauthenticated, phone + password + `device_id`, `services.login_waiter`) skips the admin-code step entirely: a waiter's *first-ever* device is trusted and approved automatically; a later login from a *different* `device_id` instead creates a **pending** (`is_approved=False`) `StaffDevice`, rejects with `403`/"Menejer tasdig'i kutilmoqda", and fires a `device_approval_requested` `Notification` to all managers. A manager clears it via `POST /api/devices/<pk>/approve/` (`StaffDeviceViewSet.approve` → `services.approve_device`, same `IsAdminStaff` gate as `revoke` below), which deactivates the waiter's previously-approved device, activates+approves the pending one, deletes and reissues the user's `Token` (kills old sessions), and force-disconnects the old device's WebSocket. **`login_waiter` looks up the `(user, device_id)` row via `filter().order_by('-id').first()`, not `get_or_create`** — a waiter can have multiple historical `StaffDevice` rows for the same `device_id` (e.g. one from the registration-code flow, one from TOFU), and `get_or_create` would raise `MultipleObjectsReturned` (500) the second time such a device re-logs in. Re-login from a *previously-approved-but-now-inactive* device (the common case: switching back to an old phone after a new one got approved) is treated as a fresh untrusted device too — `is_approved` is force-reset to `False` and it goes back through manager approval, rather than being silently reactivated as `is_active=True` while still `is_approved=True`, which would violate the one-active+approved-device-per-user constraint (`IntegrityError` → 500) since the *current* device is also active+approved.
- **One active+approved device per user**, enforced by partial unique DB constraints on `StaffDevice` scoped to `is_active=True, is_approved=True` (on both `user` and `device_id`) — the added `is_approved` clause vs. the original `is_active`-only constraint is what lets a *pending* waiter-approval row coexist alongside the still-active old device without violating uniqueness. Registering an already-approved device (the admin-code path) still silently evicts the old one (`services._evict_active_device`).
- **Revocation** (`StaffDeviceViewSet.revoke`, `IsAdminStaff`-only) is soft (`is_active=False`, history preserved via `django-simple-history`) but has real effect immediately: it deletes the user's `Token` (blocks future HTTP requests) **and** calls `realtime.force_disconnect(user_id)` to close any already-open WebSocket connection — token deletion alone wouldn't touch a connection that's already established.
- PIN attempts are rate-limited per `device_id` in Redis (5 wrong attempts → 429 for 5 minutes), independent of any Django-level auth throttling.
- **Device identity is now re-checked on every request, not just at login:** `core/authentication.py::DeviceTokenAuthentication` (the sole `DEFAULT_AUTHENTICATION_CLASSES` entry in `settings.py`, replacing plain `TokenAuthentication`) requires a matching `X-Device-ID` header naming an `is_active=True, is_approved=True` `StaffDevice` for the token's user on *every* API call — `is_staff` accounts are exempt (they never had a device), and under `manage.py test`/`TESTING` a missing header falls back to the old no-device-check behavior so tests that predate this don't need the header. A stale/wrong/missing header on a non-staff user fails the whole request with `401` — so a just-revoked device can't keep hitting other endpoints on its still-valid token until it also gets caught at login. The Channels WS handshake (`core/middleware_ws.py::TokenAuthMiddleware`) mirrors this check.

**Real-time WebSocket events** (`core/consumers.py`, `core/routing.py`, `core/realtime.py`, `core/middleware_ws.py`, wired in `config/asgi.py`): a single reusable channel at `ws/events/`, authenticated via `?token=<DRF token>&device_id=<device_id>` query params (`TokenAuthMiddleware` — Channels' built-in `AuthMiddlewareStack` only understands Django sessions, so this reimplements the same DRF `Token` + device-approval lookup as `DeviceTokenAuthentication` above, rather than adding a second auth system). Every connection joins the shared `restaurant` broadcast group plus a per-user `user_<id>` group; a blocked license closes the connection at `connect()` time (code `4402`). Any view/service pushes a message with `realtime.broadcast_event(event_type, payload)` — adding a new event type never touches the consumer, just a new call site (already used for `price_changed`, `table_status_changed`, `order_updated`, `discount_applied`, `new_print_job`, `print_job_updated`). Payloads are deliberately thin "re-fetch" signals (e.g. `{"table_id": N}`) — except `new_print_job`, which carries a full items snapshot, since the kitchen dashboard has nothing else to fetch that state from. `force_disconnect` targets the per-user group only, used by device revocation and by `approve_device` evicting a waiter's old device.
